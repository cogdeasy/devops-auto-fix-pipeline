# Stage 5: PR Creation

## Input
- `patch`, `fix_description`, `files_changed` from Stage 3
- `diagnosis` from Stage 2
- `job_name`, `build_number` from Stage 1
- `validation_build_number` from Stage 4

## Process

### 5.1 Compose PR Content

**Title**: `fix: {fix_description} [auto-fix #{build_number}]`

**Body**:
```markdown
## Auto-Fix: {job_name} Build #{build_number}

### Root Cause
{diagnosis.root_cause}

Error type: {diagnosis.error_type}
Severity: {diagnosis.severity}

### Changes Made
{fix_description}

### Files Modified
- {file1}
- {file2}

### Risk Assessment
- **Scope**: {number of files changed} file(s) modified
- **Confidence**: {confidence level from patch generation}
- **Side effects**: {assessment based on change scope}

### Validation
- Validation build: #{validation_build_number} - PASSED
- All existing tests: PASSED

### Original Failure
- Job: {job_name}
- Build: #{build_number}
- Timestamp: {failure_timestamp}

### Known Issues Referenced
{links to Confluence pages if any, or "None"}

---
*Generated by AI Auto-Fix Pipeline*
```

**Labels**: `ai-auto-fix`

### 5.2 Create the PR

**Automated mode** (github-mcp or gh CLI):
- Use `gh pr create` or github-mcp to create the PR
- Assign reviewers based on CODEOWNERS or default team
- Add the `ai-auto-fix` label

**Manual mode**:
- Output the complete title and body in a code block
- Ask: "Would you like me to run `gh pr create`, or will you create the PR manually?"
- If manual: user copies the content into their PR tool

### 5.3 Pipeline Summary

Output a final summary:
```
Pipeline Execution Summary
==========================
Reference:    AI-FIX-{timestamp}
Job:          {job_name}
Build:        #{build_number}
Error Type:   {diagnosis.error_type}
Fix:          {fix_description}
Files:        {files_changed}
Validation:   PASSED (build #{validation_build_number})
PR:           {pr_url}
Retries:      {retry_count}/3
Status:       COMPLETE
```

## Output

```
pr_url: string
pr_number: number
pipeline_summary: string
```
